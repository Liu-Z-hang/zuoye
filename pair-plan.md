# 结对编程计划

## 成员

| 成员 | 角色说明 |
| --- | --- |
| 刘子航 | 参与需求讨论、功能实现、代码复审和集成测试 |
| 陈相宇 | 参与需求讨论、功能实现、代码复审和集成测试 |

## 沟通机制

- 开发前共同确认本次任务目标、影响模块和验收方式。
- 每次编码前由驾驶员说明当前要实现的具体功能，领航员确认实现方向。
- 遇到接口、数据库、页面交互不一致时，先在群聊或 Issue 中说明问题，再决定修改方案。
- 每次 PR 合并前，两人共同确认功能是否能运行、代码是否符合项目结构。

## 冲突解决方式

- 所有功能分支都从 `develop` 创建，开发前先同步最新 `develop`。
- 如出现冲突，由当前功能分支驾驶员负责解决，领航员负责复查冲突处理是否正确。
- 不允许为了省事覆盖他人代码；存在分歧时，以需求文档和现有接口约定为准。
- 合并前必须通过 PR，由另一名成员 review 并批准。

## 分支策略

```text
main
  └── develop
        ├── feature/user-module-liuzihang
        ├── feature/seat-module-chenxiangyu
        ├── feature/order-module-liuzihang
        └── feature/pricing-module-chenxiangyu
```

- `main`：最终稳定版本，只接收来自 `develop` 的阶段性合并。
- `develop`：集成分支，用于汇总已完成并通过复审的功能。
- `feature/*`：功能分支，每个模块单独开发并通过 PR 合并。

## PR 复审规则

- 禁止直接向 `main` 推送代码。
- 禁止两人直接向同一个 feature 分支推送。
- feature 分支合并到 `develop` 前，必须由另一名成员完成复审并批准。
- `develop` 合并到 `main` 前，两人共同确认系统主要功能可运行。

## 角色互换规则

- 驾驶员负责具体代码实现、语法细节和当前逻辑。
- 领航员负责整体架构、代码风格、即时缺陷发现和下一步策略。
- 每 30-45 分钟互换一次驾驶员和领航员。
- 换人提交之间至少间隔 30 分钟。
- 每次互换记录写入 `docs/role-rotation-log.md`。
